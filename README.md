InventoryManagement API is a RESTful backend application built with ASP.NET Core 9 that provides a complete warehouse and stock management system covering products, categories, suppliers, and stock movements. The project follows Clean Architecture with Domain, Application, Infrastructure, and API layers, ensuring that business rules live entirely in the inner layers and are never coupled to database or HTTP concerns. The Domain layer defines four entities: Product, Category, Supplier, and StockMovement, all inheriting from a BaseEntity that provides Id, CreatedAt, and UpdatedAt fields. The Product entity is the central entity in the system — it holds a name, a unique SKU code, an optional description, a unit price, a current quantity, a low stock threshold, a status, and foreign keys to both a Category and a Supplier. The Product entity exposes an IsLowStock computed property that returns true when the current quantity is greater than zero but less than or equal to the low stock threshold, making it easy to identify products that need restocking. The ProductStatus enum has three values: Active for products in stock, OutOfStock for products with zero quantity, and Discontinued for products that are no longer sold; the status is updated automatically whenever the quantity changes. The Category entity is a simple named grouping with an optional description and a collection of products; deleting a category that still has products associated with it is blocked at the service level with a descriptive error. The Supplier entity stores a name, contact email, phone, and address, and supports two lifecycle states through the SupplierStatus enum: Active and Inactive; individual activate and deactivate endpoints allow toggling the status without deleting the supplier. The StockMovement entity records every change to a product's quantity and carries the quantity involved, the type of movement, an optional reason text, and the exact timestamp of the movement. The StockMovementType enum has four values: StockIn for incoming goods, StockOut for goods leaving the warehouse, Return for goods coming back from customers, and Adjustment for manual corrections that can be either positive or negative. The Domain layer also defines four repository interfaces — IProductRepository, ICategoryRepository, ISupplierRepository, and IStockMovementRepository — and an IUnitOfWork that composes all four and provides a single SaveChangesAsync method. The IProductRepository includes specialized query methods for filtering by category, filtering by supplier, and retrieving all low-stock products directly from the database, avoiding the need to load and filter everything in memory. The Application layer contains ProductService, CategoryService, SupplierService, and StockMovementService, each implementing a corresponding interface. The ProductService handles the full product lifecycle including creation with SKU uniqueness validation, updating non-stock fields, discontinuing a product, and deleting it. The StockMovementService is the most logic-heavy service in the project: it parses the movement type from the request, calculates the correct quantity delta — positive for StockIn and Return, negative for StockOut, and either sign for Adjustment — validates that the resulting quantity would not go below zero, persists the movement record, and updates the product quantity and status in the same transaction. The GetSummaryAsync method in StockMovementService aggregates data across the entire system and returns total product count, the number of products below their low-stock threshold, the number of out-of-stock products, total supplier count, total category count, and total inventory value calculated as the sum of unit price multiplied by quantity for every product. The Infrastructure layer implements all four repositories using Entity Framework Core with full Include navigation for related entities, configures AppDbContext with precise column types and constraints, sets up the IsLowStock property as an ignored computed property so EF Core does not try to map it to a column, and registers everything through a DependencyInjection extension method. The StockMovement cascade delete is configured so that when a product is deleted all its movement history is also removed, while Category and Supplier use restrict delete behavior to prevent orphaning products. The API layer contains four controllers: ProductsController, CategoriesController, SuppliersController, and StockMovementsController, none of which require authentication since this project is designed as an internal management tool where authentication is expected to be handled at the infrastructure or gateway level. The ProductsController exposes ten endpoints including a dedicated GET /api/products/low-stock route that returns only products currently below their threshold. The StockMovementsController exposes a GET /api/stock-movements/summary route that returns the full inventory summary without requiring any parameters. The GlobalExceptionMiddleware provides consistent error handling by catching KeyNotFoundException as 404, InvalidOperationException as 400, and all unexpected errors as 500, all formatted as JSON objects with an error field. The appsettings.json contains only a SQL Server connection string since this project has no authentication; the connection string points to a local instance by default and can be updated to any SQL Server environment. Running dotnet restore from the solution root installs all required NuGet packages, after which the EF Core migration must be added and applied targeting the Infrastructure project to create the four database tables. Once the application is running the Swagger UI is immediately accessible and provides a complete view of all endpoints organized by controller, making it straightforward to test the full stock management workflow from creating categories and suppliers through adding products and registering stock movements.
